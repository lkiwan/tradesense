The Pre-Authentication Layer: Visitor Experience and Product ArchitectureThe user lifecycle begins in the pre-authentication phase, where the platform functions primarily as a conversion engine. The architecture must support dynamic configuration of these challenges, as evidenced by FundedNext's evolution from "Express" models to the current "Stellar" suite.1.1.1 The Stellar Architecture (Current Standard)The "Stellar" model represents the current operational standard, characterized by a specific balance of risk parameters and reward incentives. Understanding this structure is critical for database schema design, as the backend must validate trade data against these specific constraints.The Stellar 1-Step Model is architected for speed and high risk. It operates as a single-phase evaluation, requiring a database flag that bypasses secondary evaluation stages upon success. The profit target is fixed at a demanding 10% 1, while the risk parameters are tightened significantly to a 3% Maximum Daily Loss 1 and a 6% Maximum Overall Loss.1 This creates a high-turnover environment where the system must be capable of processing failure events rapidly. The targets are stepped down—8% for Phase 1 and 5% for Phase 2 1—reducing the immediate pressure on the trader but prolonging the evaluation lifecycle. The risk parameters are relaxed to a 5% Daily Loss and 10% Overall Loss 1, necessitating a different set of threshold monitors in the risk engine. The Rule Engine: Risk Management and LogicThe "Brain" of the FundedNext operation is the Rule Engine. This is a backend service that monitors every active account in real-time to enforce the trading rules. A strong structure requires this engine to be fault-tolerant and extremely low-latency.6.1 The Drawdown Monitor (Balance-Based vs. Equity-Based)FundedNext employs a Balance-Based Drawdown for its Stellar models.2 This is a critical distinction for the risk engine's logic.Logic: The "Daily Loss Limit" is calculated based on the account balance at the start of the trading day (Server Time 00:00).Formula: Max Daily Loss = (Balance @ 00:00) - (Current Equity).Implication: If a user starts the day with $100,000 and the limit is 5% ($5,000), they breach if Equity hits $95,000. Crucially, if they profit $2,000 (Equity = $102,000), their breach level remains $95,000 (allowing for a $7,000 dip). This contrasts with "Equity-Based" or "Trailing" drawdown, which would ratchet the breach level up.Implementation: The system snapshots the balance of every account at 00:00 server time and stores it as Daily_Start_Balance. A separate process monitors Current_Equity via the Manager API. If Current_Equity < (Daily_Start_Balance - Daily_Limit), a "Breach" signal is fired.6.2 The Profit Target and Trading Day ValidatorProfit Target: The engine checks if Current_Equity >= (Initial_Balance * (1 + Target_Percent)).1Minimum Trading Days: Even if the profit target is hit, the user cannot pass until they satisfy the minimum days (e.g., 5 days for Stellar 2-Step).1 The system counts a "Trading Day" as any day where at least one trade was opened/closed. The "Pass" button in the dashboard remains disabled until Days_Active >= 5.6.3 Prohibited Strategy DetectionThe risk engine also scans trade logs for patterns indicating prohibited behavior:Arbitrage: Latency arbitrage or triangular arbitrage is detected by comparing trade execution times against feed latency logs.25Group Trading: The system analyzes IP addresses and trade correlation across different users to detect "Group Hedging" (one user goes long, another goes short).25Grid/Martingale: While generally flexible, extreme versions of these strategies (account rolling) might trigger risk alerts.7.
